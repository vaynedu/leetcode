# 567. 字符串的排列（Permutation in String）

## 题目信息

- 题号：LeetCode 567
- 标题：Permutation in String
- 难度：中等（Medium）
- 模式：滑动窗口 / 固定长度窗口

## 一句话思路

判断 `s2` 中是否存在一个长度为 `len(s1)` 的窗口，窗口字符计数与 `s1` 完全相同。

## 为什么想到这个模式

`s1` 的排列长度固定，排列只改变字符顺序，不改变字符频次。所以问题等价于：

```text
s2 是否存在一个连续窗口，使窗口长度 = len(s1)，且字符计数完全相同
```

这是固定长度滑动窗口。用 `need/window/valid` 可以避免每次比较完整 map。

选择这个模式的信号：

- 连续子串。
- 目标长度固定。
- 判断排列，也就是字符频次相等。
- 只要找到一个即可返回。

## 状态定义 / 数据结构

- `need`：`s1` 的字符频次。
- `window`：当前窗口里相关字符频次。
- `valid`：完全满足 `window[c] == need[c]` 的字符种类数。
- `needLen`：需要满足的字符种类数。
- `L, R`：窗口边界，窗口是 `[L, R)`。

当窗口长度为 `len(s1)` 且 `valid == needLen` 时，说明找到一个排列。

## 核心不变量

窗口长度达到 `len(s1)` 后，每轮都会先检查，再移除左侧字符，保证每个固定长度窗口都被判断一次。

`valid` 的更新只在“刚好相等”时发生：

- 从不满足变成满足：`valid++`。
- 从满足变成不满足：`valid--`。

不能用 `>=`，因为字符超量不是匹配。

## 手推全过程

以 `s1 = "ab", s2 = "eidbaooo"` 为例：

```text
need = {a:1, b:1}
```

| 窗口 | 状态 | valid | 结论 |
|------|------|-------|------|
| `"ei"` | 没有 a/b | 0 | 不匹配 |
| `"id"` | 没有 a/b | 0 | 不匹配 |
| `"db"` | b 满足，缺 a | 1 | 不匹配 |
| `"ba"` | a/b 都满足 | 2 | 返回 true |

窗口 `"ba"` 是 `"ab"` 的一个排列。

## Go 代码实现

```go
func checkInclusion(s1 string, s2 string) bool {
    need := make(map[rune]int)
    window := make(map[rune]int)

    for _, c := range s1 {
        need[c]++
    }

    L, R := 0, 0
    valid := 0
    needLen := len(need)

    for R < len(s2) {
        c := rune(s2[R])
        if _, ok := need[c]; ok {
            window[c]++
            if window[c] == need[c] {
                valid++
            }
        }
        R++

        if R-L == len(s1) {
            if valid == needLen {
                return true
            }
            d := rune(s2[L])
            if _, ok := need[d]; ok {
                if window[d] == need[d] {
                    valid--
                }
                window[d]--
            }
            L++
        }
    }

    return false
}
```

## 错误写法与反例

### 错误 1：`valid++` 用 `>=`

```go
if window[c] >= need[c] {
    valid++
}
```

反例：`s1 = "ab", s2 = "aa"`。

`a` 超量不代表 `b` 满足。`valid` 只能在某一类字符频次刚好等于需求时增加。

### 错误 2：窗口长度超过后才收缩

```go
if R-L > len(s1) {
    shrink()
}
```

这样会让长度大于 `len(s1)` 的窗口参与判断。排列长度必须完全相等，不能更长。

### 错误 3：找到后继续滑但忘记返回

```go
if valid == needLen {
    found = true
}
```

本题只问是否存在，找到一个就可以返回。继续滑动虽然也能做，但更容易引入后续状态错误；和 438 的“记录所有起点”要区分。

## 复杂度

| 指标 | 复杂度 |
|------|--------|
| 时间 | O(n) |
| 空间 | O(k)，k 是字符种类数；小写字母场景可视为 O(1) |

## 四问复盘

1. 这题为什么属于这个模式？
   因为排列长度固定，问题等价于在 `s2` 中找一个固定长度且字符频次相等的连续窗口。

2. 这个算法维护的不变量是什么？
   每个被检查的窗口长度都是 `len(s1)`；`valid` 表示频次完全满足的字符种类数。

3. 每一步为什么不会漏答案？
   窗口从左到右滑动，所有长度为 `len(s1)` 的子串都会被检查一次，匹配时立即返回。

4. 哪个边界最容易错，错了会怎样？
   最容易错的是把字符超量当成满足。`window[c] > need[c]` 不是匹配，会导致误判不存在的排列。
