# 146. LRU 缓存

## 题目信息

- 题号：LeetCode 146
- 标题：LRU Cache
- 难度：中等（Medium）
- 模式：设计 / HashMap + 双向链表

## 一句话思路

HashMap 负责 O(1) 定位节点，双向链表负责 O(1) 维护使用顺序：头部最近使用，尾部最久未使用。

## 为什么想到这个模式

题目要求 `get` 和 `put` 都是 O(1)，同时还要淘汰“最久未使用”的 key。

单独 HashMap 只能 O(1) 查找，不能维护最近使用顺序。单独链表能维护顺序，但查找 key 是 O(n)。组合后：

- `map[key] -> *Node`：O(1) 找到节点。
- 双向链表：O(1) 把任意节点删除并移动到头部。
- 尾部节点：O(1) 找到最久未使用节点。

## 状态定义 / 数据结构

- `cache map[int]*Node`：key 到链表节点的映射。
- `head`：虚拟头节点，`head.next` 是最近使用节点。
- `tail`：虚拟尾节点，`tail.prev` 是最久未使用节点。
- `capacity`：最大容量。

链表节点：

```go
type Node struct {
    key, val   int
    prev, next *Node
}
```

节点必须保存 `key`，因为淘汰尾部节点时还要从 map 中删除对应 key。

## 核心不变量

任意时刻：

- map 中的每个 key 都对应链表中的一个真实节点。
- 链表中的每个真实节点也都能在 map 中找到。
- 链表从头到尾按最近使用程度递减。
- `head` 和 `tail` 是哨兵，不存业务数据。

操作规则：

- `Get` 命中：节点变成最近使用，移到头部。
- `Put` 已存在：更新 value，并移到头部。
- `Put` 新 key：插入头部；若超容量，删除 `tail.prev`。

## 手推全过程

以 `capacity = 2` 为例：

| 操作 | 链表状态（头 -> 尾） | 结果 | 说明 |
|------|----------------------|------|------|
| `put(1,1)` | `[1]` | - | 1 最近使用 |
| `put(2,2)` | `[2,1]` | - | 2 最近使用 |
| `get(1)` | `[1,2]` | 1 | 1 被访问，移到头部 |
| `put(3,3)` | `[3,1]` | - | 超容量，淘汰尾部 2 |
| `get(2)` | `[3,1]` | -1 | 2 已被淘汰 |
| `put(4,4)` | `[4,3]` | - | 超容量，淘汰尾部 1 |
| `get(1)` | `[4,3]` | -1 | 1 已被淘汰 |
| `get(3)` | `[3,4]` | 3 | 3 移到头部 |
| `get(4)` | `[4,3]` | 4 | 4 移到头部 |

## Go 代码实现

```go
func Constructor146(capacity int) LRUCache {
    head := &Node{}
    tail := &Node{}
    head.next = tail
    tail.prev = head
    return LRUCache{
        capacity: capacity,
        head:     head,
        tail:     tail,
        cache:    make(map[int]*Node, capacity),
    }
}

func (this *LRUCache) Get(key int) int {
    if node, ok := this.cache[key]; ok {
        this.moveToHead(node)
        return node.val
    }
    return -1
}

func (this *LRUCache) Put(key int, value int) {
    if node, ok := this.cache[key]; ok {
        node.val = value
        this.moveToHead(node)
        return
    }

    node := &Node{key: key, val: value}
    this.cache[key] = node
    this.addToHead(node)
    if len(this.cache) > this.capacity {
        tailNode := this.tail.prev
        this.removeNode(tailNode)
        delete(this.cache, tailNode.key)
    }
}
```

## 错误写法与反例

### 错误 1：`Get` 命中后不移动到头部

```go
func (c *LRUCache) Get(key int) int {
    return c.cache[key].val
}
```

反例：`put(1), put(2), get(1), put(3)`。如果 `get(1)` 不刷新使用顺序，会错误淘汰 1；正确应淘汰 2。

### 错误 2：淘汰链表尾部但忘记删 map

```go
c.removeNode(tailNode)
```

map 里还留着旧 key，后续 `get` 会命中一个已经不在链表中的节点，破坏 map 和链表一致性。

### 错误 3：节点不保存 key

淘汰 `tail.prev` 时只能拿到节点，如果节点没有 key，就无法 O(1) 从 map 删除对应项。

## 复杂度

| 操作 | 时间 | 空间 |
|------|------|------|
| Get | O(1) | O(capacity) |
| Put | O(1) | O(capacity) |

## 四问复盘

1. 这题为什么属于这个模式？
   因为需要同时 O(1) 查找 key 和 O(1) 维护最近使用顺序，HashMap + 双向链表正好分别解决这两个需求。

2. 这个算法维护的不变量是什么？
   map 和链表中的真实节点一一对应；链表头部最近使用，尾部最久未使用。

3. 每一步为什么不会漏答案？
   每次访问或更新都会把节点移动到头部，尾部始终代表全局最久未使用节点，淘汰时直接删除它。

4. 哪个边界最容易错，错了会怎样？
   最容易错的是淘汰时忘记同步删除 map，会导致已经被淘汰的 key 仍然能被查到。
