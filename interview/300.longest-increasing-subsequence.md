# 300. 最长递增子序列

## 题目信息

- 题号：LeetCode 300
- 标题：Longest Increasing Subsequence
- 难度：中等（Medium）
- 模式：动态规划 / 二分贪心

## 一句话思路

维护 `tails[i]` 表示长度为 `i+1` 的递增子序列的最小尾值；用二分找到第一个 `>= v` 的位置替换。

## 为什么想到这个模式

LIS 可以用 O(n^2) DP：`dp[i]` 表示以 `i` 结尾的 LIS 长度。但只求长度时，可以维护每种长度的最优尾值。

尾值越小，后续越容易接上更大的数。因此对同样长度，只保留最小尾值。

## 状态定义 / 数据结构

- `tails[i]`：长度为 `i+1` 的递增子序列中，所有可能尾值的最小值。
- `len(tails)`：当前已知 LIS 长度。

对每个 `v`：

- 找第一个 `tails[i] >= v`。
- 若不存在，追加，说明 LIS 长度增加。
- 否则替换，让同长度子序列尾值更小。

## 核心不变量

`tails` 始终单调递增。

`tails[i]` 不一定是真实答案序列的一部分，但它代表长度 `i+1` 的子序列能达到的最小尾值。替换不会减少已有 LIS 长度，只会提升未来扩展机会。

## 手推全过程

`nums = [10,9,2,5,3,7,101,18]`

| v | 操作 | tails |
|---|------|-------|
| 10 | 追加 | `[10]` |
| 9 | 替换 10 | `[9]` |
| 2 | 替换 9 | `[2]` |
| 5 | 追加 | `[2,5]` |
| 3 | 替换 5 | `[2,3]` |
| 7 | 追加 | `[2,3,7]` |
| 101 | 追加 | `[2,3,7,101]` |
| 18 | 替换 101 | `[2,3,7,18]` |

答案是 `4`。

## Go 代码实现

```go
func lengthOfLIS(nums []int) int {
    tails := make([]int, 0)
    for _, v := range nums {
        i := 0
        for i < len(tails) && tails[i] < v {
            i++
        }
        if i == len(tails) {
            tails = append(tails, v)
        } else {
            tails[i] = v
        }
    }
    return len(tails)
}
```

说明：当前代码用线性查找位置，思想是 `tails`；可把查找替换成二分，达到 O(n log n)。

## 错误写法与反例

### 错误 1：严格递增用 `>` 或 `<=` 搞混

本题是严格递增，寻找的是第一个 `>= v` 的位置。若允许相等追加，`[2,2,2]` 会错误得到长度 3。

### 错误 2：认为 `tails` 就是最终子序列

`tails` 只保证长度和最小尾值，不一定是原数组中的真实 LIS 序列。题目只要求长度，所以可以。

### 错误 3：替换时以为会丢答案

替换同长度的更小尾值不会让长度变短，只会让未来更容易扩展。

## 复杂度

| 指标 | 复杂度 |
|------|--------|
| 当前实现时间 | O(n^2)，因为查找位置是线性 |
| 二分优化时间 | O(n log n) |
| 空间 | O(n) |

## 四问复盘

1. 这题为什么属于这个模式？
   因为只求 LIS 长度时，可以用每种长度的最小尾值代表最优扩展状态。

2. 这个算法维护的不变量是什么？
   `tails[i]` 是长度 `i+1` 递增子序列的最小可能尾值，且 `tails` 单调递增。

3. 每一步为什么不会漏答案？
   追加会增加长度，替换只改善同长度尾值，不会删除任何更长可能性。

4. 哪个边界最容易错，错了会怎样？
   最容易错的是严格递增条件。相等元素不能扩展长度，否则重复值会被误算进 LIS。
