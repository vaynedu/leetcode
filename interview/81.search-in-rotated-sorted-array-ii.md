# 81. 搜索旋转排序数组 II

## 题目信息

- 题号：LeetCode 81
- 标题：Search in Rotated Sorted Array II
- 难度：中等（Medium）
- 模式：二分查找 / 旋转数组 / 重复元素

## 一句话思路

和 33 一样判断哪一半有序；遇到 `nums[left] == nums[mid] == nums[right]` 时无法判断，只能收缩边界跳过重复值。

## 为什么想到这个模式

数组原本有序，只是被旋转过。虽然整体不再有序，但任意一次二分后，通常至少有一半仍然有序。只要判断目标是否落在有序半区，就能排除另一半。

本题相比 33 多了重复元素，重复会造成无法判断哪半有序：

```text
[1, 0, 1, 1, 1]
 left mid right 都可能相等
```

因此仍然是旋转数组二分，但最坏会退化到 O(n)。

## 状态定义 / 数据结构

- `left`：当前搜索区间左边界。
- `right`：当前搜索区间右边界。
- `mid`：当前中点。
- 搜索区间：闭区间 `[left, right]`。

目标：始终保证如果 `target` 存在，它一定在 `[left, right]` 中。

## 核心不变量

每轮循环都保持：`target` 如果存在，必然还在当前搜索区间内。

分三种情况：

- `nums[mid] == target`：直接返回。
- 左半有序：`nums[left] <= nums[mid]`，判断 `target` 是否在 `[nums[left], nums[mid])` 内。
- 右半有序：否则右半有序，判断 `target` 是否在 `(nums[mid], nums[right]]` 内。

重复元素特殊情况：

```text
nums[left] == nums[mid] && nums[mid] == nums[right]
```

此时无法判断哪半有序。因为已经确认 `nums[mid] != target`，且两端值和 `nums[mid]` 一样，所以 `nums[left]` 和 `nums[right]` 也不是 target，可以安全收缩边界。

## 手推全过程

以 `nums = [2,5,6,0,0,1,2], target = 0` 为例：

| left | mid | right | 值 | 判断 | 决策 |
|------|-----|-------|----|------|------|
| 0 | 3 | 6 | 2,0,2 | `nums[mid] == target` | 返回 true |

再看重复导致退化的例子：`nums = [1,1,1,1,1], target = 0`。

| left | mid | right | 值 | 判断 | 决策 |
|------|-----|-------|----|------|------|
| 0 | 2 | 4 | 1,1,1 | 三者相等，无法判断 | 收缩边界 |
| 1 | 2 | 3 | 1,1,1 | 三者相等，无法判断 | 收缩边界 |
| 2 | 2 | 2 | 1,1,1 | 三者相等，无法判断 | 收缩边界 |

最后区间为空，返回 false。重复值让二分无法排除一半，所以最坏 O(n)。

## Go 代码实现

```go
func SearchRotatedII(nums []int, target int) bool {
    if len(nums) == 0 {
        return false
    }
    left, right := 0, len(nums)-1
    for left <= right {
        mid := left + (right-left)/2
        if nums[mid] == target {
            return true
        }

        if nums[left] == nums[mid] && nums[mid] == nums[right] {
            left++
            right--
        } else if nums[left] <= nums[mid] {
            if nums[left] <= target && target < nums[mid] {
                right = mid - 1
            } else {
                left = mid + 1
            }
        } else {
            if nums[mid] < target && target <= nums[right] {
                left = mid + 1
            } else {
                right = mid - 1
            }
        }
    }
    return false
}
```

## 错误写法与反例

### 错误 1：直接套 33，不处理三者相等

```go
if nums[left] <= nums[mid] {
    ...
}
```

反例：`nums = [1,0,1,1,1], target = 0`。

`nums[left] == nums[mid]` 会让代码误以为左半有序，但目标其实在被旋转打断的位置，可能被错误排除。

### 错误 2：三者相等时只 `left++`，但忘记先判断 `mid`

```go
if nums[left] == nums[mid] && nums[mid] == nums[right] {
    left++
}
if nums[mid] == target {
    return true
}
```

如果 `mid` 正好是 target，先收缩可能错过答案。应先检查 `nums[mid] == target`。

### 错误 3：区间边界不一致

```go
for left < right {
    ...
}
```

本实现使用闭区间 `[left, right]`，循环应是 `left <= right`。写成 `<` 会漏掉最后一个候选元素。

## 复杂度

| 指标 | 复杂度 |
|------|--------|
| 时间 | 平均 O(log n)，最坏 O(n) |
| 空间 | O(1) |

## 四问复盘

1. 这题为什么属于这个模式？
   因为旋转数组局部仍有序，通常可以通过判断有序半区来二分排除候选。

2. 这个算法维护的不变量是什么？
   如果 `target` 存在，它始终留在 `[left, right]` 搜索区间内。

3. 每一步为什么不会漏答案？
   有序半区能判断目标是否落入其中；不在就安全排除。三者相等时，已确认这些重复边界值不是目标，可以收缩。

4. 哪个边界最容易错，错了会怎样？
   最容易错的是重复元素导致无法判断有序半区。若不特殊处理，可能把包含目标的一半错误排除。
